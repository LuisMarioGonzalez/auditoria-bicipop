📋 Resumen Ejecutivo

Se realizó una auditoría de seguridad web completa contra la aplicación bicipop.duckdns.org, un marketplace de bicicletas desarrollado como proyecto del Bootcamp KeepCoding XIX. La auditoría utilizó la herramienta CAI (Cybersecurity AI) con los agentes especializados Web App Pentester (agente 20), DNS/SMTP (agente 9), Bug Bounter (agente 4), Red Team (agente 12) y Retester (agente 15), combinando reconocimiento pasivo y pruebas activas de vulnerabilidades.

La aplicación presenta deficiencias de configuración que deben corregirse antes de un despliegue en producción real.

🔎 Hallazgos Detallados

CRÍTICO

H-01 — Ausencia total de cabeceras de seguridad HTTP

CWE-693 | Afecta: todos los endpoints

La aplicación no implementa ninguna de las cabeceras de seguridad HTTP recomendadas por OWASP. Esto amplía la superficie de ataque ante clickjacking, XSS, MIME sniffing y ataques de downgrade TLS.

Evidencia:

$ curl -sk https://bicipop.duckdns.org -I | grep -iE "x-frame|csp|hsts|x-content|x-xss" # Sin output — CONFIRMADO: ninguna cabecera de seguridad presente

Cabeceras ausentes:

CabeceraRiesgo por ausencia
X-Frame-OptionsClickjacking — la página puede incrustarse en iframe malicioso
Content-Security-PolicyXSS sin restricción de fuentes de scripts
Strict-Transport-SecurityDowngrade TLS, ataques MITM
X-Content-Type-OptionsMIME sniffing — el navegador puede interpretar archivos de forma insegura
X-XSS-ProtectionSin protección XSS en navegadores legacy

✅ Recomendación — nginx.conf

add_header X-Frame-Options "DENY" always; add_header X-Content-Type-Options "nosniff" always; add_header X-XSS-Protection "1; mode=block" always; add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always; add_header Content-Security-Policy "default-src 'self';" always;
CRÍTICO

H-02 — Dominio vulnerable a Email Spoofing

CWE-290 | Afecta: registros DNS del dominio

El dominio no tiene registro SPF y el registro DMARC existe pero está completamente vacío. Cualquier atacante puede enviar emails suplantando @bicipop.duckdns.org sin restricción técnica.

Evidencia:

$ dig TXT bicipop.duckdns.org | grep spf # Sin output — CONFIRMADO: registro SPF ausente $ dig TXT _dmarc.bicipop.duckdns.org _dmarc.bicipop.duckdns.org. 5 IN TXT "" # CONFIRMADO: DMARC vacío — sin política de protección $ dig MX bicipop.duckdns.org bicipop.duckdns.org. 5 IN MX 50 bicipop.duckdns.org. # MX apunta al propio dominio — configuración inusual

✅ Recomendación — registros DNS

  • Añadir registro SPF: bicipop.duckdns.org. TXT "v=spf1 -all"
  • Configurar DMARC con política de rechazo: _dmarc.bicipop.duckdns.org. TXT "v=DMARC1; p=reject; rua=mailto:admin@bicipop.duckdns.org"
ALTO

H-03 — Credenciales Sentry expuestas en HTML público

CWE-200 | Afecta: / (página principal)

Las credenciales de configuración de Sentry (plataforma de monitoreo de errores) se exponen en texto plano en las meta tags HTML de cada página, accesibles para cualquier visitante.

Evidencia:

$ curl -sk https://bicipop.duckdns.org/ | grep -o "sentry-public_key=[a-f0-9]*" sentry-public_key=df3081142f1b087d672e527c0dec2b7c # Datos completos expuestos en meta baggage: sentry-environment=production sentry-public_key=df3081142f1b087d672e527c0dec2b7c sentry-org_id=4510834727321600 sentry-trace_id=9b242c5d53289a6c0c063b882defe4e5

✅ Recomendación

  • Inicializar Sentry desde el servidor (server-side), nunca en el cliente
  • Usar variables de entorno del servidor para el DSN
  • Nunca exponer el Org ID ni el Trace ID en el HTML público

H-04 — Exposición de datos de usuario e IDs internos sin autenticación

CWE-200 | Afecta: /search (payload RSC) | Detectado por: Agente 12 (Red Team)

El endpoint público /search devuelve en su payload RSC (React Server Components) la estructura completa de cada producto, incluyendo el ID interno de usuario (CUID2), el username y la ubicación, sin autenticación. Esto permite enumerar usuarios reales y, combinado con H-07, habilita ataques IDOR usando esos IDs internos.

Evidencia — payload expuesto:

{ "id": "cmqp6xavu000004l1cdxbyyq4", "title": "Prueba producto", "price": 1234, "userId": "cmpmrudon000gotwffpkzsq4y", // ID interno de usuario "location": "Cadiz", // Geolocalización "username": "Antonia", // Username real "imageUrl": "https://gqwarqgiibymlrxlklsh.supabase.co/..." } # Usuarios reales enumerados a partir del payload: Antonia -> cmpmrudon000gotwffpkzsq4y David -> cmqf3sshd000001mjjpi2abzb Paco -> cmpmrudon000fotwfzs8foj7j Pilar -> cmpmrudon000hotwf9jm2ijzv

✅ Recomendación

  • No incluir el userId interno en los payloads públicos
  • Exponer solo un identificador público no correlacionable si es imprescindible
  • Aplicar control de acceso a los endpoints que reciban esos IDs (mitiga también H-07)
MEDIO

H-05 — Meta robots=noindex inconsistente con entorno de producción

CWE-215 | Afecta: / (todas las páginas)

Todas las páginas incluyen <meta name="robots" content="noindex"/>, pero la configuración de Sentry indica sentry-environment=production. Esta contradicción sugiere una falsa sensación de seguridad por oscuridad.

Evidencia:

<meta name="robots" content="noindex"/> <meta name="baggage" content="sentry-environment=production,..."/>

✅ Recomendación

  • Si la app es pública: eliminar noindex
  • Si debe ser privada: implementar autenticación real, no depender de oscuridad

H-06 — Exposición de infraestructura Supabase (verificada)

CWE-200 | Afecta: /search (payload RSC) | ✔ Superficie anónima verificada como protegida

El payload RSC de /search expone la URL del backend Supabase, revelando el Project ID (gqwarqgiibymlrxlklsh) y la ruta del bucket público de imágenes. Este identificador es público por diseño en Supabase, por lo que la exposición en sí es de bajo impacto.

Evidencia:

https://gqwarqgiibymlrxlklsh.supabase.co/storage/v1/object/public/products/

✔ Verificación realizada — ambas vías de explotación están protegidas:

$ curl -si ".../storage/v1/object/list/products" HTTP/2 400 -> {"error":"Bucket not found"} # no se puede enumerar el bucket $ curl -si ".../rest/v1/users?select=*" HTTP/2 401 -> UNAUTHORIZED_MISSING_API_KEY # la REST API exige la anon key

Riesgo residual (no explotado): queda por confirmar si la anon key del bundle permitiría consultar tablas con RLS desactivado — fuera del alcance de las pruebas anónimas.

✅ Recomendación

  • Verificar que las políticas RLS (Row Level Security) están activas en todas las tablas
  • Configurar el bucket como privado con URLs firmadas si las imágenes no deben ser públicas
  • No exponer detalles del backend en los payloads del cliente

H-07 — Productos accesibles sin autenticación (IDOR potencial)

CWE-639 | Afecta: /products/:id

Los productos son accesibles sin autenticación mediante su ID en la URL. Se confirmó HTTP 200 con contenido completo del producto.

Evidencia:

$ curl -sk https://bicipop.duckdns.org/products/cmpmrun9t003yotwfs060rsop \ -o /dev/null -w "HTTP %{http_code}" HTTP 200 # CONFIRMADO — página accesible sin autenticación

✅ Recomendación

  • Verificar si hay datos de productos que requieran autenticación
  • Implementar control de acceso basado en roles si procede
